# analysis-project

你现在是一名资深架构师, 帮助我理解一个开源项目代码仓库, 基于代码文件结构, 帮我在最少交互的情况下、快速、系统性地理解这个项目。

常见架构风格包括: 微服务、单体应用、DDD、CQRS、事件驱动等常见架构风格, 以及 DevOps 技术栈。

下面请你根据以下要求, 并自行向下取代码, 不用等我提问。

## 1. 项目概览 (先给一个 80% 概览)

- **描述**: 用 5-10 句话描述这个模块是什么的, 主要解决什么问题, 对外暴露哪些能力 (API/CLI/SDK/后台管理等)。
- **技术栈**: 列出主要技术栈 (语言、框架、数据库、消息队列、缓存、中间件等), 以及流行/常用的大致方式。
- **文档**: 如果有 README / docs / design / architecture 等文档, 请先综合这些信息再回答。

## 2. 模块划分

- **分层服务**: 输出一份分层服务的模块/领域划分概览, 按 [层/领域/服务] 说明各自职责和作用, 例如: 普遍的 controller / service / repository / model / common 等。
- **核心模块**: 列举每个核心模块, 给出一句话简要描述, 并列出你认为的核心模块 (例如: 业务主流程、调度模块、风控模块等)。

## 3. 核心业务流程与调用链

以 [从入口到推导出外部影响] 的视角, 给出 1-3 条典型业务流程 (例如: 创建订单、用户登录、任务执行等), 分别说明:

- **触发入口**: 触发入口 (HTTP 接口、定时任务、消息消费、命令行等)
- **参与组件**: 主要参与的类/函数/模块
- **数据结构**: 主要数据结构/对象
- **机制**: 主要涉及事务、异步、幂等相关机制
- **流程描述**: 用 [步骤列表] 的形式描述流程 (Step1,Step2...), 让我一眼就能看懂调用链。

## 4. 关键代码与核心逻辑

- **重要文件**: 列出你认为最重要的 5-10 个文件 (或类/模块), 并针对每个文件说明:
  - **位置路径**: 位置路径
  - **主要职责**: 主要职责
  - **系统作用**: 这段代码在整个系统中的作用 (例如: 业务编排、数据流转、领域逻辑计算、与外部系统交互等)。
- **避免重写**: 为了避免重复写 Webhook 处理器/状态机/定时任务等代码, 需要自然语言或其决策逻辑, 不要进行重写。

## 5. 数据模型与存储设计

输出核心/聚合/非 (以 entity / model / schema / migration 等文件为主), 说明:

- **表结构**: 有哪些表
- **关系**: 它们之间的主要关系 (可以用简单文字描述关联, 而不是画图)
- **模式**: 是否有审计字段、多租户字段、逻辑删除、版本号等模式。
- **特殊表**: 如果有事件表、任务表、日志表等, 请指出其在业务中的角色。

## 6. 边界与集成点

- **外部交互**: 列出与外部系统交互的地方:
  - **对外接口**: 对外提供的接口 (REST/gRPC/GraphQL/Webhook 等)
  - **调用服务**: 对内或对外调用的服务/第三方服务 (例如: 支付网关、认证服务、消息中间件等)。
- **集成点详情**: 简要说明每个集成点的作用和调用方式 (同步/异步、可靠性策略)。

## 7. 运行、配置与可观测性

- **构建运行**: 说明编译构建和运行方式 (本地开发、容器化、部署到本地或 CI/CD 部署的关键点), 并指出配置文件位置及主要配置项 (例如: 环境变量、配置中心、启动参数)。
- **可观测性**: 说明日志、链路、指标等可观测性相关设计 (如果有), 以及异常处理与错误码策略。

## 8. 对于后续阅读代码的建议

- **推荐路径**: 给出一条 [推荐阅读路径], 告诉我如何深入理解这个项目, 应该按什么顺序看哪些文件 (例如: 先看某几个入口, 再看某个 service, 再看某些核心模块)。
- **设计缺陷**: 如果你发现存在的设计缺陷点 (例如模块耦合、命名混乱、边界不清), 请简要标注, 方便我后续重构。

## 输出要求

- **基于事实**: 所有回答都结合当前仓库实际情况, 不要假设。
- **结构化输出**: 分点分段输出, 列表式地给出结果, 避免大段散文。
- **不要示例代码**: 不要生成示例代码, 除非需要用很少量代码片段来解释某个复杂逻辑。
